## ðŸ’¾ DATABASE DESIGN & DATA MODELING

### **Core Database Schema**
Design PostgreSQL schema using Prisma with these key models:

**Users Table**: Store user authentication and profile data
- Fields: id (UUID), email (unique), name, avatar_url, created_at, updated_at
- Relationships: One-to-many with organization_members, accounts, sessions

**Organizations Table**: Business entities that own data and subscriptions
- Fields: id (UUID), name, slug (unique), subscription_tier, billing_email, created_at
- Relationships: One-to-many with members, integrations, insights, reports

**Integrations Table**: Connected platforms and their credentials
- Fields: id (UUID), organization_id, platform (shopify/stripe/etc), access_token (encrypted), refresh_token, status, last_sync_at
- Security: Encrypt tokens with AES-256, store expiration times

**Data_Points Table**: Time-series data from all integrations
- Fields: id (UUID), integration_id, metric_type, value (decimal), metadata (JSON), date_recorded
- Indexing: Composite index on (integration_id, date_recorded) for fast queries
- Partitioning: Partition by month for performance

**Insights Table**: AI-generated business insights
- Fields: id (UUID), organization_id, type (trend/anomaly/recommendation), title, description, impact_score (1-10), is_read
- Purpose: Store generated insights for dashboard display

### **Data Relationships & Constraints**
- Users can belong to multiple organizations through organization_members junction table
- Each integration belongs to one organization but can have many data points
- Insights are organization-specific and ranked by impact score
- Implement soft deletes for user data (GDPR compliance)
- Add created_at/updated_at timestamps to all tables

### **Performance Optimization**
- Create indexes on frequently queried fields (organization_id, date_recorded, metric_type)
- Use PostgreSQL views for complex dashboard queries
- Implement connection pooling with Prisma
- Set up read replicas for analytics queries

## ðŸ”Œ API DESIGN & BACKEND DEVELOPMENT

### **RESTful API Structure**
Design clean, consistent API endpoints following REST principles:

**Authentication Endpoints**:
- POST /api/auth/register - User registration with email/password
- POST /api/auth/login - User login and session creation  
- GET /api/auth/session - Get current user session
- POST /api/auth/logout - Destroy user session

**Organization Management**:
- GET /api/organizations - List user's organizations
- POST /api/organizations - Create new organization
- GET /api/organizations/[id] - Get organization details
- PUT /api/organizations/[id] - Update organization settings
- DELETE /api/organizations/[id] - Delete organization (soft delete)

**Dashboard Data**:
- GET /api/organizations/[id]/dashboard?period=30d - Get dashboard metrics and charts
- GET /api/organizations/[id]/metrics/[type]?period=7d&granularity=day - Get specific metric data
- GET /api/organizations/[id]/insights?limit=10&unread=true - Get AI insights

**Integration Management**:
- GET /api/integrations/available - List available integration platforms
- POST /api/integrations/connect - Connect new integration via OAuth
- GET /api/organizations/[id]/integrations - List organization integrations
- PUT /api/integrations/[id]/sync - Trigger manual data sync
- DELETE /api/integrations/[id] - Disconnect integration

### **Authentication & Authorization**
Implement secure authentication using NextAuth.js:
- Support Google OAuth, email/password login, magic links
- JWT tokens for API authentication with 24-hour expiration
- Role-based access control (owner/admin/member/viewer)
- API middleware to verify user permissions for organization access
- Rate limiting: 100 requests per minute per user

### **Data Processing Pipeline**
Build background job system for data synchronization:
- Queue system using Bull Queue with Redis for job management
- Scheduled jobs for hourly/daily data sync from integrations
- Error handling with exponential backoff retry logic
- Job monitoring dashboard for troubleshooting sync issues
- Webhook handlers for real-time updates from platforms

### **API Response Format**
Standardize all API responses with consistent structure:
```
Success: { success: true, data: {...}, meta: { total, page, limit } }
Error: { success: false, error: { code, message, details } }
```
